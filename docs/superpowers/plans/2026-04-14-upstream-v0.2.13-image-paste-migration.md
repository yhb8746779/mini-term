# 上游 v0.2.13 图片粘贴移植计划

> **For Claude Code / agentic workers:** 按本文档顺序逐步实现；不要直接整包 cherry-pick 上游 `v0.2.13` release commit，因为当前 fork 已经在 `v0.2.27` 线上继续演进，直接套版本提交容易把本地增强和版本号回滚掉。

## 背景

本仓库已经包含上游 `v0.2.12` 的源码基线，并且在其后继续演进了大量本地增强（AI provider 状态、项目最近会话排序、PSReadLine / zsh 检测修复、CJK 宽字符修复、CSI 3J scrollback 保留、macOS Ctrl 键兼容等）。

对比上游 `dreamlonglll/mini-term` 在 `v0.2.12 -> v0.2.13` 之间的源码变化后，可移植且最值得补充的能力只有一组：

- **终端图片粘贴**
  - `Ctrl+Shift+V` 粘贴剪贴板图片
  - 右键菜单粘贴剪贴板图片
  - Windows 下通过 Win32 剪贴板 API 兜底 PinPix 等非标准截图格式
  - 启动时自动清理超过 24 小时的临时截图文件

## 总体实现思路

### 前端

`pasteToTerminal()` 增强为三段式：

1. 检测剪贴板是否包含图片
2. 如果有图片：
   - 先尝试调用后端 `read_clipboard_image`
   - 成功则把临时 PNG 路径写入 PTY
   - 失败则回退发送 `Alt+V`（`\x1bv`）
3. 如果没有图片：
   - 保持现有文本粘贴行为

### 后端

新增一个 Tauri command：

- `read_clipboard_image() -> Result<String, String>`

行为：

- Windows：
  - 优先从 Win32 剪贴板读取 `CF_DIB`
  - 再退回读取 `CF_BITMAP`
  - 保存成 `%TEMP%/mini-term-clipboard/clip-<timestamp>.png`
  - 返回文件路径字符串
- 非 Windows：
  - 直接返回错误字符串，前端会自动回退到 `Alt+V`

此外，在应用启动时清理 `%TEMP%/mini-term-clipboard` 下超过 24 小时的 PNG 文件。

- [ ] **Step 1: 扩展 clipboard import**

把：

```ts
import { readText, writeText } from '@tauri-apps/plugin-clipboard-manager';
```

改成：

```ts
import { readText, readImage, writeText } from '@tauri-apps/plugin-clipboard-manager';
```

- [ ] **Step 2: 右键菜单无选中时复用 `pasteToTerminal()`**

当前 `src/utils/terminalCache.ts` 的 context menu 分支里，无选中时还是：

```ts
readText().then((text) => {
  if (text) {
    invoke('write_pty', { ptyId, data: text });
    term.focus();
  }
});
```

改为：

```ts
void pasteToTerminal(ptyId).finally(() => {
  term.focus();
});
```

**理由：**

- 避免快捷键和右键走两套分叉逻辑
- 后续图片粘贴只维护一处
- 保留 `term.focus()` 行为

- [ ] **Step 3: 新增 `clipboardHasImage()` helper**

在 `pasteToTerminal()` 上方添加：

```ts
async function clipboardHasImage(): Promise<boolean> {
  try {
    await readImage();
    return true;
  } catch {
    // Tauri 插件可能读不到某些图片格式
  }

  try {
    const items = await navigator.clipboard.read();
    return items.some((item) => item.types.some((t) => t.startsWith('image/')));
  } catch {
    // 浏览器 Clipboard API 在某些环境/权限下可能不可用
  }

  return false;
}
```

**为什么双检测：**

- `readImage()` 能覆盖标准图片剪贴板
- `navigator.clipboard.read()` 可以补一层 MIME 检测
- 上游就是双路径探测，能降低误判“没有图片”的概率

- [ ] **Step 4: 改造 `pasteToTerminal()`**

把当前的纯文本实现：

```ts
export async function pasteToTerminal(ptyId: number): Promise<void> {
  const text = await readText();
  if (text) await enqueuePtyWrite(ptyId, text);
}
```

改成：

```ts
export async function pasteToTerminal(ptyId: number): Promise<void> {
  if (await clipboardHasImage()) {
    try {
      const path: string = await invoke('read_clipboard_image');
      await enqueuePtyWrite(ptyId, path);
      return;
    } catch {
      // Windows 非标准格式兜底失败，退回 Alt+V 让 AI 工具自己读图
    }

    await enqueuePtyWrite(ptyId, '\x1bv');
    return;
  }

  const text = await readText().catch(() => null);
  if (text) await enqueuePtyWrite(ptyId, text);
}
```

**关键约束：**

- 仍然使用现有 `enqueuePtyWrite()`，不要直接绕过写队列
- 图片分支成功时写入“路径字符串”，不要自动加引号
- `readText()` 改成 `catch(() => null)`，避免 clipboard 读取异常把整个粘贴流程炸掉

- [ ] **Step 5: 前端静态检查**

通过标准：

- 无新 TS error

### Task 3: Rust 后端增加图片剪贴板模块

**文件：**

- [ ] **Step 1: 新建 `clipboard.rs`**

基本结构如下：

```rust
#[cfg(windows)]
mod win {
    // Win32 剪贴板读取实现
}

pub fn cleanup_old_clipboard_images() {
    // 清理 temp_dir()/mini-term-clipboard 下超 24h 的 PNG

#[tauri::command]
pub fn read_clipboard_image() -> Result<String, String> {
    #[cfg(windows)]
    {
        let path = win::read_clipboard_to_png()?;
        Ok(path.to_string_lossy().into_owned())
    }

    #[cfg(not(windows))]
    {
        Err("仅支持 Windows 平台".into())
    }
}
```

- [ ] **Step 2: Windows 模块实现 `CF_DIB -> CF_BITMAP` 双兜底**

实现要求：

- 优先尝试 `CF_DIB`
- 再尝试 `CF_BITMAP`
- 支持至少：
  - `24-bit`
  - `32-bit`
- 输出统一转成 RGBA PNG

建议直接参考上游 `8845bc9` 的实现结构，核心 API 包括：

- `OpenClipboard`
- `CloseClipboard`
- `GetClipboardData`
- `IsClipboardFormatAvailable`
- `GlobalLock`
- `GlobalUnlock`
- `GlobalSize`
- `GetObjectW`
- `CreateCompatibleDC`
- `SelectObject`
- `GetDIBits`
- `DeleteDC`

**实现原则：**

- 错误信息用中文字符串即可，和本项目现有风格一致
- `unsafe` 代码块范围尽量收窄
- 成功 / 失败路径都必须关闭剪贴板

- [ ] **Step 3: 保存为临时 PNG**

保存规则：

- 目录：`std::env::temp_dir().join("mini-term-clipboard")`
- 文件名：`clip-<unix_millis>.png`

保存 helper 要确保：

- 目录不存在时自动创建
- 返回完整文件路径

- [ ] **Step 4: 实现旧文件清理**

`cleanup_old_clipboard_images()` 要求：

- 启动时调用一次
- 只清理 `mini-term-clipboard` 目录

## 实现时的风险点

### 1. 不要覆盖当前 `terminalCache.ts` 的本地增强

这个文件里已有：

- Unicode11 宽字符修复
- CSI 3J 拦截
- WebGL 降级
- macOS WKWebView Ctrl 修复
- 写队列

因此不能直接用上游文件覆盖，必须手工插入最小变更。

### 2. 右键菜单当前不是复用 `pasteToTerminal()`

这是本地代码与上游的重要差异。

如果只改 `pasteToTerminal()` 而没改 context menu 分支，最终会出现：

- `Ctrl+Shift+V` 可以贴图
- 右键菜单仍然只能贴文本

所以 **Task 1 Step 2 是必做项**。

### 3. 非 Windows 平台不能崩

`read_clipboard_image()` 必须用 `#[cfg(windows)]` / `#[cfg(not(windows))]` 包起来。

要求：

- 非 Windows 能编译
- 非 Windows 调用该命令时只返回错误，不 panic
- 前端捕获错误后自动退回 `Alt+V`

### 4. `Alt+V` 回退是必要兜底，不要删除

原因：

- 即便检测到“剪贴板里有图片”
- 也仍可能遇到 Tauri 插件读不到
- 或 Win32 API 读失败

此时仍然需要：

```ts
await enqueuePtyWrite(ptyId, '\x1bv');
```

这样 Codex / Claude 等支持图片粘贴的 TUI 工具还能继续工作。

### 5. 路径粘贴不要自动加引号

上游实现是直接把路径字符串写入 PTY，不包双引号。

保持这个行为，避免：

- shell 里多出意外转义
- 和现有拖拽文件插路径的交互习惯不一致
